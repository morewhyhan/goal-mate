## 增量需求和存量修改的写法

### 增量需求

适用于全新功能、全新模块、全新页面或全新业务流程。

写法重点：

- 先写业务背景，再定义新模块边界。
- 明确用户角色、业务目标、入口、出口和主流程。
- 完整列出字段清单、业务规则、验收标准。
- 补充原型链接或 HTML 原型位置。
- 如果涉及 API、数据库、权限、E2E，需要在需求中显式说明。

### 存量修改

适用于对已有功能、已有页面、已有接口、已有字段或已有规则做调整。

写法重点：

- 先说明当前行为，再说明目标行为。
- 用变更清单标注新增、修改、删除、废弃内容。
- 明确兼容性影响：旧数据、旧接口、旧流程、旧测试是否受影响。
- 字段清单只写受影响字段，但必须说明与现有字段的关系。
- 验收标准必须覆盖变更前后的关键差异。

## 需求规格必须包含

- 业务背景。
- 需求类型：增量需求或存量修改。
- 模块范围。
- 原型链接或 HTML 原型位置。
- 字段清单。
- 业务规则。
- 验收标准。
- 非目标。
- 待确认问题。

新需求请复制 `template-feature.md`。
