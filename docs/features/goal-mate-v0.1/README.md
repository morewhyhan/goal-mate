# Goal Mate v0.1 需求规格

版本：v0.1  
状态：当前开发基线  
需求类型：增量需求，全新产品第一版  
事实来源：`docs/shit/shit.md`、当前产品讨论、`docs/designs/goal-mate-mcp-console.html` 作为表现参考  
核心结论：原型只做 UI 参考，需求以目标推进秘书的核心链路为准。

## 1. 产品一句话

Goal Mate 是一个 AI 目标秘书：用户只输入目标结果和每日反馈，AI 自动规划、提醒、诊断、调整和控风险，持续推动用户把想法落成现实结果。

更短表达：

```text
我只说想要什么、每天做得怎么样；它负责中间判断，并每天告诉我下一步。
```

## 1.1 产品哲学

Goal Mate 的核心工作假设是：

```text
行动力问题，本质上优先被当作认知和决策问题处理。
```

用户没有行动，不应被简单理解为“不自律”。系统必须先判断决策链哪里没有做到位：

| 判断 | 含义 | 系统应该做什么 |
| --- | --- | --- |
| 方向不真 | 这件事不是用户真正想要的，可能只是外部期待 | 不继续硬催，先帮助用户重审目标、暂停或重新选择 |
| 难度过高 | 目标真实，但当前行动步子太大 | 降低难度，把行动拆成更小、更容易启动的一步 |
| 提示不足 | 用户想做，也知道怎么做，但关键时刻没有被提醒 | 在风险点前给出具体提示，让行动重新进入控制范围 |
| 路径错误 | 用户在努力，但行动没有补齐关键条件 | 调整路径，换成更能产生现实结果的行动 |

因此，产品不是让用户管理复杂计划，而是让 AI 接管中间过程：

```text
输入：用户想要的现实结果。
过程：AI 澄清目标、拆条件、判断风险、主动追问、温和推动、持续调整。
输出：现实世界中可验证的变化。
```

例如某个现实结果达到可验证指标、某项能力达到验收标准、某个输出形成稳定节奏、某个项目进入可交付状态。用户只需要关心自己想达到什么效果，以及现在下一步做什么；方法论、节奏、追问和调整由 AI 在后台处理。

## 2. v0.1 产品边界

v0.1 不是完整版，也不是普通打卡工具。它要验证一个核心假设：

```text
即使只支持一个主目标，AI 也必须完整经历：理解目的 -> 倒推条件 -> 生成成功标准 -> 拆阶段节点 -> 每日推进 -> 未完成诊断 -> 路径调整 -> 阶段复盘。
```

因此 v0.1 的边界是：

| 类型 | 内容 |
| --- | --- |
| 必须做 | 单主目标完整推进闭环 |
| 必须做 | Web Console 作为第一版主要入口 |
| 必须做 | 邮箱密码登录、注册、会话门禁和退出登录 |
| 必须做 | Agent 对话作为目标澄清、规划、调整入口 |
| 必须做 | Markdown Logs 作为年/季/月/周/日推进记录 |
| 必须做 | 模型配置、提醒配置、数据导出等基础设置 |
| 不做 | 多人协作、企业 OKR、复杂项目管理 |
| 不做 | 复杂日历/邮箱自动化 |
| 不做 | 移动 App、小程序、自部署 MCP 的完整实现 |
| 不做 | 让用户自己学习和维护复杂 OKR 表格 |

## 3. 产品形态

v0.1 首先做 Web。

```text
Web Console
  -> Login：登录、注册、进入个人工作区
  -> Today：今天下一步
  -> Goals：目标拆解与进度，只读
  -> Logs：Markdown 推进记录，可编辑
  -> Agent：专业对话工作台
  -> Settings：模型、提醒、日志、数据、隐私配置
```

机器人、微信、飞书、移动端和自部署能力是后续形态，但 v0.1 的设计必须为它们预留核心数据和接口边界。

## 4. 用户

主要用户：

```text
创业者、独立开发者、学生/备考人群、自由职业者、副业人群、内容创作者、有长期目标但执行容易中断的人。
```

用户心理：

```text
不想维护一个复杂工具。
不想学习 OKR 和项目管理术语。
希望 AI 理解目标、拆路、提醒、诊断和复盘。
打开产品时，只想知道现在下一步该做什么。
```

## 5. v0.1 必须成立的核心链路

```text
用户输入目的
  ↓
Agent 澄清真正目标和成功标准
  ↓
Goal Engine 生成目标推理卡
  ↓
后台生成隐式 OKR 和关键结果
  ↓
倒推必要条件与充分条件组合
  ↓
拆成阶段节点、周目标和今日行动
  ↓
Today 展示当前唯一下一步
  ↓
用户完成或未完成
  ↓
Agent 诊断动机、能力、提示、路径问题
  ↓
更新目标、条件、行动、提醒和日志
  ↓
生成日/周/月/目标周期记录
```

这条链不能砍。v0.1 可以限制目标数量和平台数量，但不能降低 AI 的推理深度。

## 6. 页面清单

| 页面 | 目的 | 是否 v0.1 必须 |
| --- | --- | --- |
| Login | 让用户创建账户或登录，并进入自己的工作区 | 是 |
| Today | 告诉用户今天下一步做什么 | 是 |
| Goals | 只读展示目标、KR、条件、阶段、进度 | 是 |
| Logs | 以 Markdown 层级记录年/季/月/周/日推进过程 | 是 |
| Agent | 通过对话澄清、规划、诊断、调整和控制系统 | 是 |
| Settings | 配置模型、提醒、日志、数据、隐私、集成 | 是 |
| Reviews | 独立复盘中心 | 否，v0.1 合并到 Logs |
| Inbox | 外部上下文收件箱 | 否 |
| Self-host Console | MCP 和本地服务控制台 | 否 |

## 8. 总体验收

| 编号 | 验收标准 |
| --- | --- |
| V01-AC-1 | 用户能通过 Agent 输入一个模糊目的，并得到目标澄清问题 |
| V01-AC-2 | 系统能生成目标推理卡，包含成功标准、KR、必要条件、充分条件、当前缺口 |
| V01-AC-3 | Today 只展示当前最重要的一步，并说明它补齐哪个关键条件 |
| V01-AC-4 | Goals 能只读展示目标树、KR、阶段计划、进度和当前推进状态 |
| V01-AC-5 | Logs 能按年/季/月/周/日层级展示并编辑 Markdown 记录 |
| V01-AC-6 | Agent 有历史记录、消息区和固定输入框，能读取目标与日志上下文 |
| V01-AC-7 | 用户未完成行动时，系统能诊断动机、能力、提示或路径问题 |
| V01-AC-8 | Settings 能配置默认模型、DeepSeek、提醒、日志路径、数据导出和隐私 |
| V01-AC-9 | 复盘以 Markdown 日志方式沉淀，能回答本周期补齐了什么条件、下周期推进什么 |
| V01-AC-10 | Agent 和 Scheduler 不只是固定提醒，而能根据目标状态自主决定追问内容、提示时机和下一步干预 |
| V01-AC-11 | Meta-Cognition 能根据干预效果更新对用户动机、能力、风险点、提示时机和表达偏好的判断 |
| V01-AC-12 | Memory / Logs / Review 沉淀的关键判断必须充分、必要、因果明确、语言清晰、可验证或可证伪 |
| V01-AC-13 | Today、Agent 和 Scheduler 产生的反馈必须进入同一套控制回合语义，串起干预、反馈、诊断、状态更新、日志和元认知 |
| V01-AC-14 | 元认知必须形成闭环：ControlLoopEpisode 提供证据，Meta-Cognition 生成或评估假设并形成 policy_delta，下一次 Intervention Planner 必须消费该变化 |
| V01-AC-15 | 涌现效果必须通过连续回合证明：系统能基于多次反馈增强、削弱、修正或过期旧假设，并改变下一次干预策略 |
| V01-AC-16 | AI 自我优化必须形成闭环：旧干预被证伪后，系统必须记录 AI 上一次判断错在哪里，以及下一次推理规则 |
| V01-AC-17 | 下一次 Planner 必须消费 AI 自我优化规则，让下一次问题、风险判断或行动安排发生可观察变化 |
| V01-AC-18 | 登录注册必须形成用户可感知闭环：未登录不能进入 Dashboard，用户能注册、登录、退出，并且所有私有数据只能归属当前 session 用户 |
