# F8 后续版本边界

## 1. 为什么保留边界说明

`docs/shit/shit.md` 里已经明确：最终产品会有技术用户自部署版和普通用户云端版，核心引擎只有一套。v0.1 不实现所有形态，但必须避免把核心模型写死在 Web 原型里。

## 2. 后续形态

| 形态 | 用户 | 价值 | 版本 |
| --- | --- | --- | --- |
| Web Console | 所有早期用户 | 快速验证目标推进闭环 | v0.1 |
| 微信/飞书/Telegram 机器人 | 普通用户和团队用户 | 主动提醒和自然回复 | v0.2+ |
| Mobile App / 小程序 | 普通用户 | 更强提醒和日常使用体感 | v0.3+ |
| Self-hosted Agent | 技术用户、隐私用户 | 数据自控、模型自配、MCP | v0.3+ |
| Cloud App | 普通用户 | 开箱即用、多端同步 | v0.4+ |

## 3. 核心引擎必须独立

Goal Engine 不能依赖具体 UI。它应该只关心：

```text
目标澄清
隐式 OKR
条件倒推
行动生成
未完成诊断
路径调整
复盘生成
结构化输出校验
```

Web、机器人、App、自部署只是入口不同。

## 4. 后续机器人能力

机器人主要用于：

| 能力 | 说明 |
| --- | --- |
| 每日提醒 | 主动问今天行动 |
| Check-in | 用户直接回复完成/没完成 |
| 未完成诊断 | 用自然对话判断原因 |
| 日志写入 | 自动把反馈写入 Logs |
| 快速调整 | 用户说“明天改到晚上提醒” |

机器人不负责展示完整 Goals 和 Logs，复杂查看仍回到 Web。

## 5. 后续自部署能力

自部署版应该包含：

```text
Local Web UI
Local API
MCP Server
SQLite / PostgreSQL
Scheduler
Model Provider 配置
数据导出/备份
Docker Compose 启动
```

v0.1 Settings 和 Data Model 必须为这些能力预留字段。

## 6. 后续普通用户形态

普通用户最终更可能喜欢移动端或微信入口，因为提醒和反馈发生在日常环境中。但第一版选择 Web，是为了更快打磨核心链路和开发效率。

## 7. 后续不变原则

| 原则 | 说明 |
| --- | --- |
| 核心引擎一套 | 不为每个平台重写目标逻辑 |
| 用户不学方法论 | OKR、条件倒推、行为模型都在后台 |
| 行动优先 | 所有入口最终都服务下一步行动 |
| 高风险确认 | 外部执行、删除、目标重大调整必须确认 |
