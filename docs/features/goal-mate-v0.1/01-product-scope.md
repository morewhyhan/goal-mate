# F0 产品边界规格

## 1. 背景

用户并不是缺少待办事项，而是缺少一个能长期理解目标、倒推路径、每天推进、失败后诊断并持续调整的 AI 秘书。

普通 Todo 工具只记录任务；传统 OKR 工具要求用户自己懂 OKR；普通提醒工具只会催促。Goal Mate 的核心是让 AI 在后台理解方法论，让用户只面对目标、下一步、反馈和复盘。

## 2. v0.1 定位

v0.1 是 Web Console 形态的单主目标完整推进闭环。

它不是：

| 不是 | 原因 |
| --- | --- |
| 普通 Todo | 不只列任务，必须解释任务和目标条件的关系 |
| 打卡工具 | 不以连续天数为核心，而以关键条件被补齐为核心 |
| 企业 OKR | 不要求用户填写组织级 OKR 表格 |
| 项目管理工具 | 不做复杂看板、资源排期和多人协作 |
| 纯聊天机器人 | 对话结果必须结构化保存并推动目标状态变化 |

它是：

```text
基于隐式 OKR、目的倒推、充分必要条件和行为诊断的 AI 目标推进秘书。
```

更底层地说，v0.1 是一个基于老三论的目标推进控制系统：

```text
系统论：把用户围绕目标的状态、资源、约束、行动和反馈看成一个整体系统。
信息论：通过对话、日志、Check-in 和记忆降低目标推进的不确定性。
控制论：通过今日行动、提醒、诊断和复盘持续修正推进路径。
```

前台体验是秘书式对话，后台结构是状态观察、偏差诊断、干预控制和反馈学习。

Goal Mate 不把“没有行动”直接归因于意志力。产品中的默认判断是：如果用户没有行动，优先说明当前目标、路径、难度或提示系统还没有被设计好。系统要帮助用户把决策做清楚，而不是用生硬催促压迫用户。

用户输入的是想要达到的现实结果，例如可量化结果、能力水平、作品输出、项目进度或收入验证；系统输出也必须回到现实结果。中间的计划、节奏、拆解、提示和路径调整，原则上由 AI 自主处理，不把复杂决策再推回给用户。

## 3. v0.1 必须做

| 能力 | 说明 |
| --- | --- |
| 目标输入 | 用户自然语言输入目标或目的 |
| 目的澄清 | Agent 追问真正目标、成功标准、时间边界 |
| 目标推理卡 | AI 生成目标理解、成功标准、KR、条件、当前缺口 |
| 隐式 OKR | 后台维护 Objective、Key Results、Initiatives，不强迫用户填写 |
| 条件倒推 | 输出必要条件、充分条件组合、当前最关键缺口 |
| 阶段拆解 | 形成阶段节点、周目标、今日行动 |
| Today 下一步 | 用户打开后只知道现在最该做什么 |
| 未完成诊断 | 判断动机、能力、提示、路径问题 |
| Logs | 自动沉淀年/季/月/周/日 Markdown 记录，并允许编辑 |
| Agent 对话 | 用户通过对话调整目标、询问原因、生成计划 |
| Settings | 配置模型、提醒、日志、数据、隐私 |

## 4. v0.1 不做

| 能力 | 放到后续原因 |
| --- | --- |
| 移动 App / 小程序 | 第一版先验证核心链路，Web 先跑通 |
| 微信/飞书机器人完整接入 | 机器人是重要入口，但不影响核心数据模型先成立 |
| 多人协作 | 会显著增加权限、组织和冲突复杂度 |
| 复杂日历/邮箱读取 | 属于外部上下文增强，不是第一版核心 |
| 自动执行外部动作 | 风险高，必须等权限模型成熟 |
| 企业 OKR 报表 | 产品不是企业管理系统 |
| 知识卡片系统 | 本产品服务行动，不服务知识收藏 |

## 5. 第一版用户

| 用户 | 典型目标 | 关键需求 |
| --- | --- | --- |
| 独立开发者 | 做产品、比赛项目、开源项目 | 把模糊目标变成每日推进 |
| 学生/备考 | 考试、技能、证书 | 每天知道学什么，没做时能调整 |
| 内容创作者 | 文章、视频、公开作品 | 保持输出节奏并复盘影响力 |
| 健康管理用户 | 减重、运动、早睡 | 行动足够小，提醒和反馈真实有效 |
| 自由职业/副业 | 产品、客户、收入 | 识别关键验证节点，避免忙错方向 |

## 6. 产品规则

| 编号 | 规则 |
| --- | --- |
| F0-R1 | v0.1 可以只支持一个 current_focus_goal，但该目标的推理链必须完整 |
| F0-R2 | 用户不需要懂 OKR，但系统必须能维护隐式 OKR |
| F0-R3 | 所有功能必须服务规划或推进，不服务这两者的第一版不做 |
| F0-R4 | Agent 的对话必须能转成结构化目标、行动、日志或设置变更 |
| F0-R5 | 目标页展示的是推进结构，不展示冗余解释和用户早已确认过的“为什么” |
| F0-R6 | 日志是推进记录，不是知识库，不做灵感收藏和 MOC |
| F0-R7 | 设置项必须真实影响系统行为，不允许假按钮 |
| F0-R8 | 所有核心功能必须进入“观察状态 - 诊断偏差 - 干预调整”的闭环 |
| F0-R9 | 页面可以简化，但不能切断目标、条件、行动、反馈和复盘之间的关系 |
| F0-R10 | 用户不需要为每个目标手动配置复杂催办规则，AI 必须根据目标状态自主决定追问、提示和干预 |
| F0-R11 | 催办语气必须温和、正常、具体，不允许使用强制、羞辱或机械施压 |
| F0-R12 | 当目标不再是用户真实想要的结果时，系统应该建议重审或暂停，而不是继续催促 |

## 7. 验收标准

| 编号 | Given | When | Then |
| --- | --- | --- | --- |
| AC-F0-1 | 用户输入“我想在 8 周内完成一个可验证成果” | Agent 分析目标 | 系统先追问成功标准和当前条件，而不是直接列任务 |
| AC-F0-2 | 用户已有一个主目标 | 打开 Today | 页面只突出一个下一步行动 |
| AC-F0-3 | 用户查看 Goals | 目标已生成 | 能看到目标、KR、条件、阶段和进度，不需要操作 |
| AC-F0-4 | 用户查看 Logs | 有推进记录 | 能按年/季/月/周/日层级查看和编辑 Markdown |
| AC-F0-5 | 用户连续没有执行某个行动 | Agent 或 Scheduler 介入 | 系统先诊断方向、难度、提示或路径问题，再决定是否调整行动、提醒或目标 |
