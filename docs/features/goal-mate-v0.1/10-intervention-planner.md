# F9 Intervention Planner 规格

## 1. 模块定位

Intervention Planner 是 Goal Mate 的主动干预决策器。

它不是提醒模板，也不是固定时间的闹钟规则。它负责在每一次主动联系用户之前，基于目标、KR、条件、Today、日志、最近反馈和用户模型，判断：

```text
今天要不要问
什么时候问
问什么
怎么提醒风险点
行动要不要降难度
用户没回复时怎么处理
什么时候建议重审或暂停目标
```

Scheduler 只负责触发时间窗口；Intervention Planner 负责决定干预内容和干预策略。

## 2. 输入

| 输入 | 说明 |
| --- | --- |
| Goal | 当前目标、时间范围、成功标准和当前状态 |
| KeyResult | 可验证结果、当前进度和证据 |
| GoalCondition | 必要条件、充分条件组合、当前缺口 |
| StagePlan | 当前阶段重点和阶段风险 |
| DailyAction | 今日行动、完成标准、最小启动和低精力备选 |
| Checkin | 最近完成、部分完成、没做、跳过的反馈 |
| Diagnosis | 最近方向、难度、提示、路径诊断 |
| Review | 周期性条件变化和下周期重点 |
| Markdown Logs | 用户手写记录和系统区块 |
| Settings | 免打扰时间、每日最大触达次数、读取权限 |
| Meta-Cognition | 用户动机、能力、风险点、提示时机、表达偏好判断，以及 AI 自我优化规则 |

## 3. 输出

Intervention Planner 每次必须输出一个可审计的干预决策。

| 字段 | 说明 |
| --- | --- |
| `intervention_type` | clarify / prompt / nudge / reduce_difficulty / risk_warning / review / pause_review |
| `target_goal_id` | 作用于哪个目标 |
| `target_condition_id` | 作用于哪个关键条件，可为空 |
| `risk_point` | 当前最需要控制的风险点 |
| `question_or_message` | 准备发给用户的一句话或一个问题 |
| `expected_feedback` | 希望用户怎么反馈，例如 done / partial / not_done / answer |
| `fallback_action` | 用户状态差时的最小补救动作 |
| `reasoning_summary` | 为什么这次这样干预 |
| `verification_signal` | 下次用什么反馈验证这次判断是否有效 |

## 3.1 AI-first Policy Planner

Intervention Planner 的主路径必须是 AI-first：AI 根据当前目标、日志、最近反馈、元认知和系统设置，生成本次干预策略。

代码层不应该把“怎么问、怎么催、怎么调整”写死。代码层只固定决策契约、质量门禁、审计字段和 fallback：

| 层 | 固定什么 | 不固定什么 |
| --- | --- | --- |
| Policy Prompt | 输出字段、语气边界、可验证要求 | 具体问题、具体风险判断、具体干预方式 |
| Quality Gate | 必须充分、必要、具体、可验证、非鸡汤 | 不替 AI 预设唯一正确答案 |
| Fallback Rule | 模型不可用或输出不合格时保底 | 不作为正常主路径 |
| Audit | `planner_source`、`policy_version`、`fallback_reason` | 不把审计内容展示成用户负担 |

`planner_source` 必须能区分：

| 值 | 含义 |
| --- | --- |
| `ai_policy` | AI 生成并通过质量门禁 |
| `fallback_rule` | 模型不可用、输出不可解析或质量门禁未通过时使用保底规则 |

质量门禁必须拒绝以下输出：

- 只有“加油、坚持、你可以的”这类泛鼓励。
- 没有明确风险点。
- 没有说明为什么这次这样问。
- 没有下次可验证信号。
- 语气强制、生硬或人格评价。

## 3.2 Planner 对 AI 自我优化的消费

Planner 不只消费“用户为什么没做”的判断，也必须消费“AI 上一次为什么没判断好”的规则。

当 Meta-Cognition 提供 `ai_self_optimization.next_thinking_rule` 时，Planner 必须先执行这条规则，再决定干预内容。

| 自我优化信号 | Planner 下一次必须怎么变 |
| --- | --- |
| 上次证据不足 | 先问一个能区分方向、难度、提示、路径的问题 |
| 上次误判为难度 | 先检查风险点或提示缺口，不继续默认降难度 |
| 上次误判为提示 | 先检查行动仓位，不继续增加提醒 |
| 上次直接安排动作无效 | 暂停排新计划，先降低不确定性 |

可证伪标准：

```text
如果旧干预被证伪后，下一次 Planner 仍使用同样的问题、同样的推理顺序和同样的动作安排，则 AI 自我优化没有生效。
```

## 4. 四类诊断

未行动时，Intervention Planner 必须优先判断四类问题：

| 类型 | 判断问题 | 干预策略 |
| --- | --- | --- |
| 方向 | 这件事是不是用户真正想要的现实结果 | 重审目标、澄清意义、建议暂停或换目标 |
| 难度 | 当前行动是不是太大、太难、太抽象 | 降低难度、拆成最小启动动作 |
| 提示 | 用户想做也能做，但关键时刻没有触发 | 调整提醒时机、在风险点前提示 |
| 路径 | 行动发生了，但没有补齐关键条件 | 调整行动、阶段重点或条件判断 |

这四类不是问卷，而是 AI 的后台判断。用户界面只应该看到自然语言问题或下一步行动。

## 5. 风险控制

Goal Mate 必须默认假设目标推进会遇到失败、偏差、风险点和失控时刻。

Intervention Planner 要像仓位管理一样控制单次行动暴露：

| 风险控制项 | 说明 |
| --- | --- |
| 行动仓位 | 今天行动不能大到一次失败就击穿用户信心 |
| 风险点 | 识别最容易脱离控制的时刻，例如环境诱因、疲惫、拖延、情绪波动 |
| 提前提示 | 在风险点出现前给出具体预案 |
| 止损动作 | 如果原行动失败，给一个仍能保住系统连续性的最小动作 |
| 失败解释 | 失败是系统反馈，不是人格评价 |
| 下一次验证 | 每次调整都必须能被下一次反馈验证或证伪 |

## 6. 用户没回复

无响应不是简单地继续加频率。

| 情况 | 处理 |
| --- | --- |
| 偶发无响应 | 等待下一个正常触达点，不加压 |
| 连续无响应 | 降低问题复杂度，只问一个最小问题 |
| 对某目标长期无响应 | 判断方向不真或路径错误，建议重审 |
| 提醒时间长期无响应 | 调整提醒时机，而不是提高频率 |
| 用户明确反感 | 降频，改变表达方式，必要时暂停该目标干预 |

## 7. 语气边界

干预语气必须温和、正常、具体。

允许：

```text
今天先别重新规划，把这一小步做完就行。
如果风险点已经出现，先执行替代动作，不要直接进入默认高风险行为。
今天偏了，但还没坏，晚上补一个最小动作。
```

不允许：

```text
你必须完成。
你不应该拖延。
请严格执行计划。
你又失败了。
```

## 8. 验收标准

| 编号 | Given | When | Then |
| --- | --- | --- | --- |
| AC-F9-1 | Scheduler 到达提醒时间 | 生成主动消息前 | 必须先形成干预决策，而不是套固定模板 |
| AC-F9-2 | 用户连续没完成行动 | Planner 判断下一步 | 必须区分方向、难度、提示、路径问题 |
| AC-F9-3 | 用户存在明确风险点 | Planner 生成消息 | 必须给出风险点前的具体提示或止损动作 |
| AC-F9-4 | 用户长期无响应 | Planner 调整策略 | 不得盲目提高频率，必须降复杂度、调时或重审目标 |
| AC-F9-5 | Planner 输出判断 | 后续 Check-in 或 Review 发生 | 必须能验证或证伪这次干预判断 |
| AC-F9-6 | Meta-Cognition 提供 `ai_self_optimization.next_thinking_rule` | Planner 生成下一次干预 | 必须先按该规则修正推理顺序，而不是复用旧问题 |
