# F4 Logs 规格

## 1. 模块定位

Logs 是 Goal Mate 的推进记录层。它借鉴 Obsidian 式 Markdown 和双向引用思想，但目的不是知识管理，而是沉淀行动推进过程。

## 2. 核心原则

```text
记录服务于推进，不服务于收藏。
```

因此不做 MOC、灵感箱、知识卡片、复杂标签系统。用户如果有灵感，应该马上和 Agent 对话，让它变成目标、条件、行动或日志，而不是单独收藏。

## 3. 层级结构

日志按时间层级组织：

```text
Logs
  2026
    Q3
      2026-07
        W27
          2026-07-01.md
```

左侧展示要像文件树：

| 层级 | 示例 | 说明 |
| --- | --- | --- |
| Year | 2026 | 年度目标和年报 |
| Quarter | Q3 | 季度目标和季报 |
| Month | 2026-07 | 月度推进和月报 |
| Week | W27 | 周计划和周报 |
| Day | 2026-07-01.md | 日行动、反馈和日记 |

## 4. Markdown 内容

### 4.1 日志模板

```md
# 2026-07-01

## 今日主目标

- 目标：
- 当前 KR：
- 当前关键条件：

## 今日行动

- 行动：
- 完成标准：
- 最小启动：

## 执行反馈

- 结果：done / partial / not_done
- 原因：
- 调整：

## 自由记录


```

### 4.2 日志中的控制信息

当普通年/季/月/周/日日志来自 Agent / QQ / Today 的行动反馈时，它需要在原有日志层级内承载控制闭环。控制、诊断、风险和调整都只是正常日志里的内容字段，不是新的日志类别。

日日志最低结构：

```md
# YYYY-MM-DD 日志

## 今日事实

## 多目标进展

## 未完成事项

## 偏差与失控点

## 未完成原因诊断

## 风险控制策略

- 风险类型：
- 控制动作：
- 后续验证信号：

## 明日计划调整

## 用户模型更新

## Agent 策略更新
```

缺少“未完成原因诊断”“风险控制策略”“用户模型更新”或“Agent 策略更新”时，该日志不能证明 Goal Mate 的行动控制闭环成立。

其中“风险控制策略”不能只写一句泛泛提醒。它至少要说清楚：

| 项 | 含义 |
| --- | --- |
| 风险类型 | 当前失控更像方向不真、任务过大、提示缺失、路径错误、现实意外、反馈质量差、多目标冲突还是低能量 |
| 控制动作 | 系统下一次具体怎么控制，例如暂停、降级、前置提醒、路径重建、止损、只问一个问题 |
| 后续验证信号 | 下一次要看什么证据，证明控制真的影响了行动结果 |

日志里的控制信息还要防止这些核心失效：

| 风险 | 日志里必须留下的证据 |
| --- | --- |
| 降低难度变成降低目标 | 今日动作可以缩小，但年/季/月/周目标和 KR 不能被偷偷改小 |
| 用户敷衍反馈 | 记录反馈质量，不把一句模糊回复当成充分事实 |
| 提醒疲劳 | 记录是否减少同日任务数和提醒频率，而不是越没做越多催 |
| 日志污染 | 只写事实、诊断、控制动作和验证信号，不写鼓励废话 |
| AI 自我优化跑偏 | 记录上一次判断是否被反馈支持，以及下一次推理规则 |
| 结果不可验证 | 行动必须有完成标准，KR 必须有可验证目标值 |
| Web/QQ 状态不同步 | QQ 反馈必须能回写同一套目标、行动、日志和复盘数据 |
| 因果误判 | 不能把一次偶然完成直接当成稳定规律，必须等后续证据支持 |

### 4.2 周志模板

```md
# 2026 W27

## 本周目标

## 本周补齐的条件

## KR 变化

## 未完成原因

## 下周重点

```

### 4.3 月志/季志/年志

月志、季志、年志不是并列按钮，而是文件树中的父级节点。它们汇总下级日志，保留 Markdown 可编辑性。

## 5. 自动引用

系统可以自动把结构化内容写入 Markdown：

| 来源 | 写入内容 |
| --- | --- |
| Goal | 目标标题、Objective、KR |
| DailyAction | 今日行动、完成标准、最小启动 |
| Checkin | 完成情况、未完成原因 |
| Diagnosis | 诊断类别、调整建议 |
| Review | 周/月/年复盘摘要 |

### 5.1 OKR 与长期日志联动

OKR 不只存在于 Goals 页面。Goals 是结构化只读视图，Logs 是长期推进证据。

日志必须能表达：

```text
年度 OKR
  -> 季度 KR
    -> 月度推进模块
      -> 周最高优先级
        -> 日日动控制日志
```

每个层级的日志职责：

| 层级 | 必须体现 |
| --- | --- |
| Year | 年度 Objective、年度 KR、季度入口 |
| Quarter | 本季度 KR、阶段重点、月度入口 |
| Month | 本月推进模块、KR 变化、周入口 |
| Week | 本周最高优先级、负载取舍、每日入口 |
| Day | 今日行动、偏差、诊断、风险控制、明日调整 |

日偏差必须能反向影响明日计划、周重点或更高层计划。否则日志只是记录，不是控制系统。

## 6. 编辑规则

| 规则 | 说明 |
| --- | --- |
| 用户可直接编辑 Markdown | 用户拥有最终文本控制权 |
| 空日志可直接开始 | 如果还没有日志，用户可以从 Logs 页面创建今日日志 |
| 自动写入不能覆盖用户手写内容 | 使用区块或追加方式 |
| 系统要保留保存状态 | 例如已保存、保存中、冲突 |
| 删除日志要确认 | 防止误删长期记录 |

## 7. 验收标准

| 编号 | Given | When | Then |
| --- | --- | --- | --- |
| AC-F4-1 | 用户打开 Logs | 有多个年份和季度 | 左侧显示层级树和连线，而不是平铺卡片 |
| AC-F4-2 | 用户点击某日日志 | 页面加载 | 右侧显示 Markdown 编辑器 |
| AC-F4-3 | 用户完成 Today 行动 | 系统保存 | 当日日志追加行动和反馈 |
| AC-F4-4 | 周期结束 | 系统生成 | 周志/月志能汇总下级日志内容 |
| AC-F4-5 | 用户通过 QQ/Today 连续反馈至少两周 | 系统生成日志 | Logs 至少形成 14 篇日日志、2 篇周志，并能看出年度 OKR、季度 KR、月度模块、周重点和今日行动的联动 |
| AC-F4-6 | 用户没完成行动 | Agent 写入日志 | 日志必须记录偏差、原因诊断、风险控制、明日调整、用户模型更新和 Agent 策略更新 |
| AC-F4-7 | 用户连续两周出现不同类型偏差 | 系统生成正常年/季/月/周/日日志 | 日志必须能指明每类风险、对应控制动作和后续验证信号，不能把所有失败都归为“没坚持”，也不能新增独立日志类型 |
