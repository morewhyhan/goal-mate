# F1 系统控制模型规格

## 1. 模块定位

Goal Mate 的底层产品模型不是普通任务管理，而是基于老三论的目标推进控制系统。

```text
系统论：把用户的目标推进过程看成一个动态系统。
信息论：通过对话、日志、反馈和记忆降低目标推进的不确定性。
控制论：通过每日行动、提醒、诊断和复盘不断修正系统状态。
```

前台表达可以是“AI 目标秘书”，后台实现必须是“状态观察 + 偏差诊断 + 干预控制 + 反馈学习”。

## 2. 总体控制回路

```text
用户目的
  ↓
目标状态建模
  ↓
关键条件与成功标准
  ↓
当前状态观察
  ↓
偏差计算
  ↓
原因诊断
  ↓
干预策略
  ↓
用户行动或无行动
  ↓
反馈记录
  ↓
模型更新
  ↓
下一次干预
```

产品里对应为：

```text
Agent 对话
  -> Goal Reasoning Card
  -> Objective / KR / Condition
  -> Stage Plan
  -> Today Action
  -> Check-in
  -> Diagnosis
  -> Review
  -> Memory / Logs
```

## 3. 系统边界

### 3.1 被控制对象

被控制对象不是用户的人格，也不是用户的一切生活，而是：

> **用户围绕某个目标的推进过程。**

系统只干预和目标推进直接相关的内容：

```text
目标是否清楚
成功标准是否可验证
关键条件是否被补齐
今日行动是否发生
未完成原因是什么
提醒是否有效
路径是否需要调整
周期复盘是否产生新判断
```

### 3.2 控制器

控制器是 AI Agent + Goal Engine。

它负责：

```text
理解用户输入
抽取目标状态
生成目标推理卡
维护隐式 OKR
选择今日行动
诊断未完成原因
调整提醒和行动
生成复盘洞察
更新长期记忆
```

### 3.3 执行器

执行器不是系统本身，而是用户。

AI 不能替用户完成身体行动、学习、写作或开发，但可以改变用户下一次行动发生的概率。

可控手段包括：

```text
问什么问题
什么时候提醒
提醒说什么
今天只推哪一步
行动拆多小
失败后如何调整
是否建议暂停或换目标
```

## 4. 关键要素清单

| 要素 | 系统含义 | 产品对象 | 主要来源 |
| --- | --- | --- | --- |
| 目的 | 用户真正想改变的方向 | purpose_summary | Agent 对话 |
| 目标状态 | 期望达到的状态 | Goal / ObjectiveState | 目标推理卡 |
| 成功标准 | 判断目标是否推进的证据 | success_signals / KR | Agent 对话、复盘 |
| 关键条件 | 达成目标必须满足的条件 | GoalCondition | 目的倒推 |
| 当前状态 | 用户现在已经具备什么 | condition status / KR progress | Check-in、日志 |
| 当前缺口 | 当前最影响推进的缺失条件 | current_gap | Goal Engine |
| 阶段重点 | 当前周期主要补齐什么 | StagePlan | Goal Engine |
| 今日行动 | 当前最小控制输出 | DailyAction | Today |
| 完成标准 | 行动是否完成的判据 | done_when | DailyAction |
| 最小启动动作 | 降低启动阻力的动作 | minimum_step | DailyAction |
| 低精力备选 | 状态差时仍可推进的动作 | fallback_action | DailyAction |
| 行动反馈 | 执行结果 | Checkin | Today / Agent |
| 未完成原因 | 偏差成因 | Diagnosis | Check-in |
| 提醒策略 | 触发行动的控制参数 | ReminderRule | Settings / Scheduler |
| 长期记忆 | 系统对用户模式的估计 | MemoryItem | 对话、日志、复盘 |
| 复盘判断 | 周期性模型更新 | Review | Review Engine |

## 5. 需要监控什么

### 5.1 目标层监控

| 监控项 | 说明 | 触发干预 |
| --- | --- | --- |
| 目标是否清楚 | 是否能说清楚期望状态 | 不清楚时追问成功标准 |
| 目标是否真实重要 | 是否持续获得用户注意力 | 长期无响应或抗拒时重审目标 |
| 成功标准是否可验证 | 是否能判断完成或推进 | 不可验证时重写 KR |
| 多目标是否冲突 | 时间和精力是否被多个目标争抢 | 必须给出取舍建议 |

### 5.2 条件层监控

| 监控项 | 说明 | 触发干预 |
| --- | --- | --- |
| 必要条件是否完整 | 是否遗漏关键达成条件 | 复盘或卡住时重建条件 |
| 条件状态是否变化 | missing、partial、satisfied 是否更新 | Check-in 后更新证据 |
| 当前缺口是否仍成立 | 当前推进重点是否还正确 | 连续未完成或反馈异常时换路 |
| 行动是否对应条件 | 今日行动是否补齐当前缺口 | 不对应时替换行动 |

### 5.3 行为层监控

| 监控项 | 说明 | 触发干预 |
| --- | --- | --- |
| 是否完成今日行动 | done、partial、not_done、skipped | 非 done 触发诊断 |
| 行动难度是否过高 | 耗时、抽象度、启动成本 | 拆小、给最小动作 |
| 用户精力是否不足 | 用户反馈疲惫、忙、状态差 | 启用低精力备选 |
| 连续未完成次数 | 是否形成阻塞模式 | 连续 3 次进入 path 诊断 |

### 5.4 提醒层监控

| 监控项 | 说明 | 触发干预 |
| --- | --- | --- |
| 提醒是否被响应 | 用户是否回复或打开 | 无响应时不盲目加频率 |
| 提醒时间是否合适 | 是否经常在错误时间触发 | 调整时间段 |
| 提醒话术是否有效 | 是否让用户知道下一步 | 改成具体行动和完成标准 |
| 提醒频率是否过高 | 是否造成打扰 | 降频或合并提醒 |

### 5.5 信息层监控

| 监控项 | 说明 | 触发干预 |
| --- | --- | --- |
| 关键字段是否缺失 | 目标、KR、条件、行动是否缺字段 | Agent 必须先补信息 |
| 信息是否过期 | 目标、计划、提醒是否已不符合现状 | 标记 stale 并重建 |
| 证据是否足够 | AI 判断是否有来源 | 低置信度时表达为当前判断 |
| 日志是否沉淀 | Check-in 是否进入日报、周报、月报 | 缺失时自动补写系统块 |

## 6. AI 可以怎么干预

| 干预类型 | 适用情况 | 具体动作 | 目标 |
| --- | --- | --- | --- |
| 澄清 | 信息不足 | 只问一个关键问题 | 降低信息不确定性 |
| 结构化 | 用户表达混乱 | 转成目标、KR、条件、行动 | 建立可控制状态 |
| 拆小 | 行动太难 | 缩小耗时、降低启动门槛 | 提高行动发生概率 |
| 替换 | 行动不对应当前缺口 | 换成更直接补齐条件的行动 | 提高路径有效性 |
| 调时 | 提醒不合适 | 改提醒时间、频率、渠道 | 提高触发有效性 |
| 重构 | 路径判断错误 | 重建目标推理卡或条件列表 | 修正系统模型 |
| 取舍 | 目标冲突 | 选择 current_focus_goal | 避免平均用力 |
| 暂停 | 目标不再重要 | 建议暂停并等待用户确认 | 避免无效控制 |
| 复盘 | 周期结束 | 总结条件变化和下一阶段重点 | 更新长期策略 |

## 7. 老三论和产品模块的映射

| 理论 | 产品中的体现 | 对应模块 |
| --- | --- | --- |
| 系统论 | 用户、目标、条件、行动、反馈、环境构成动态系统 | Goals、Today、Logs、Settings |
| 信息论 | 对话和日志把高熵表达压缩成结构化状态 | Agent、Memory、AI Interaction |
| 控制论 | 每日行动、提醒、诊断和复盘形成反馈控制 | Scheduler、Check-in、Diagnosis、Review |

## 8. 秘书式交流方法

AI 的对话不是闲聊，而是秘书式控制接口。

### 8.1 七类必要对话

| 对话类型 | 目的 | 示例 |
| --- | --- | --- |
| 澄清对话 | 补齐目标信息 | “如果 30 天后算有效，必须发生什么？” |
| 简报对话 | 告诉用户当前状态 | “今天只推进减重目标里的运动条件。” |
| 派工对话 | 给出今日行动 | “今天走路 40 分钟，做完告诉我累不累。” |
| 反馈对话 | 获取执行结果 | “完成、部分完成、没做，选一个就行。” |
| 诊断对话 | 找偏差原因 | “没做更像是不想做、太难，还是提醒不对？” |
| 调整对话 | 改行动或路径 | “明天不做完整默写，只默写一段。” |
| 复盘对话 | 更新系统判断 | “这周真正补齐的是饮食稳定性，运动还不稳定。” |

### 8.2 对话边界

AI 每次对话必须尽量满足：

```text
一次只推进一个关键点。
不用术语教育用户。
不把失败归因于人格。
不堆长篇解释。
能落到状态更新、行动、诊断或复盘。
重要判断必须说明依据或标记为当前判断。
```

## 9. 页面和模块职责

| 页面 / 模块 | 在控制系统中的角色 | 用户应该感受到 |
| --- | --- | --- |
| Agent | 信息采集入口和控制决策入口 | 我说人话，它接住并推进 |
| Today | 当前唯一控制输出 | 我现在只需要做这一步 |
| Goals | 系统状态图 | 我有哪些目标、条件和进度 |
| Logs | 反馈和记忆沉淀 | 系统知道过去发生了什么 |
| Settings | 控制参数 | 我能配置模型、提醒、数据和隐私 |
| Scheduler | 执行触发器 | 它会在合适时间来找我 |
| Review | 模型更新器 | 系统知道接下来该改什么 |

## 10. 工程约束

| 编号 | 规则 | 优先级 |
| --- | --- | --- |
| F1-R1 | 所有 DailyAction 必须能追溯到 GoalCondition | P0 |
| F1-R2 | 所有 Check-in 必须更新行动、条件、KR 或诊断中的至少一项 | P0 |
| F1-R3 | 连续未完成不能只继续提醒，必须进入诊断或路径调整 | P0 |
| F1-R4 | Agent 不能只回复文本，关键对话必须产生结构化状态变更或明确不变更原因 | P0 |
| F1-R5 | Reminder 不只是通知，必须指向当前行动或复盘问题 | P0 |
| F1-R6 | Review 不能只统计完成率，必须更新条件、路径或下一阶段判断 | P0 |
| F1-R7 | UI 不展示老三论术语，除非在文档或高级解释中 | P1 |

## 11. 验收标准

| 编号 | Given | When | Then |
| --- | --- | --- | --- |
| AC-F1-1 | 用户输入模糊目标 | Agent 响应 | 先追问成功标准或当前缺口，而不是直接列待办 |
| AC-F1-2 | 用户打开 Today | 存在 active goal | 只展示一个能补齐当前缺口的主行动 |
| AC-F1-3 | 用户反馈没做 | 系统处理 Check-in | 必须记录 diagnosis，并给出调整方向 |
| AC-F1-4 | 用户连续 3 次未完成 | Agent 或 Scheduler 介入 | 必须进入 path 级诊断，不能继续原样提醒 |
| AC-F1-5 | 用户查看 Goals | 有目标数据 | 能看到目标、KR、条件、阶段和进度之间的关系 |
| AC-F1-6 | 用户查看 Logs | 有 Check-in | 日志能还原系统如何判断和调整 |
