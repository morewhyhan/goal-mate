# F1 核心目标推进链路规格

## 1. 模块定位

核心目标推进链路是 Goal Mate 的产品灵魂。第一版可以功能少，但不能砍掉这条链。

```text
目的 -> 成功标准 -> 必要条件 -> 充分条件组合 -> 阶段节点 -> 当前最小目标 -> 今日行动 -> 执行反馈 -> 诊断调整 -> 复盘记录
```

## 2. 目标推理卡

每个 active 目标必须有一张当前有效的目标推理卡。

| 字段 | 必填 | 用户是否必须直接看到 | 说明 |
| --- | --- | --- | --- |
| raw_purpose | 是 | 可见 | 用户原始表达 |
| interpreted_goal | 是 | 可见 | AI 理解后的目标 |
| horizon | 是 | 可见 | 时间范围，例如 2026-07-01 到 2026-09-01 |
| success_signals | 是 | 可见 | 怎么算真的推进或完成 |
| objective | 是 | 可见但不强调术语 | 隐式 Objective |
| key_results | 是 | 可见 | 证明目标推进的关键结果 |
| necessary_conditions | 是 | 可见 | 没有它目标难以成立 |
| sufficient_condition_set | 是 | 可见但要简洁 | 哪些条件组合满足后目标大概率达成 |
| current_gap | 是 | 可见 | 当前最该补齐的缺口 |
| current_stage | 是 | 可见 | 当前阶段节点 |
| today_action | 是 | Today 可见 | 今天最小行动 |
| evidence | 是 | Agent 可解释 | 推理依据和上下文来源 |
| version | 是 | 后台 | 版本追溯 |

## 3. 隐式 OKR

用户可以看到 KR，但不需要学习 OKR。系统内部必须维护：

```yaml
objective:
  title: "2026 暑假完成身体、英语、内容、项目四条主线推进"
  horizon: "2026-07-01..2026-09-01"

key_results:
  - title: "体重从 165 斤降到接近 130 斤"
  - title: "新概念英语二完成默写并达到熟练"
  - title: "形成稳定日更 3000 字内容输出"
  - title: "软件杯和 Goal Mate 项目持续推进并产出可演示结果"

initiatives:
  - title: "每日简单饮食、两小时走路、一分钟平板、早睡早起"
  - title: "走路时背单词，两个月默写新概念英语二"
  - title: "每日写 3000 字并沉淀到公众号/推特/B 站"
  - title: "推进软件杯和 Goal Mate 的关键开发任务"
```

## 4. 条件倒推

AI 不能把用户目标直接变成任务列表。必须先倒推：

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| 必要条件 | 没有它目标基本无法达成 | 每天能形成稳定运动窗口 |
| 假设条件 | 当前看起来必要，后续可调整 | 两小时走路适合同时背单词 |
| 支撑条件 | 提高成功概率但不是绝对必要 | 热力图反馈连续性 |
| 充分条件组合 | 多个条件同时满足后，大概率达成目标 | 饮食控制 + 日常运动 + 睡眠稳定 + 每日记录 |

## 5. 阶段拆解

每个目标要被拆成：

```text
目标周期
  -> 阶段节点
  -> 周目标
  -> 今日行动
```

每个阶段节点必须回答：

| 问题 | 说明 |
| --- | --- |
| 这个阶段补齐哪个条件 | 例如先建立稳定节奏 |
| 怎么判断阶段完成 | 例如连续 7 天完成最小行动 |
| 当前风险是什么 | 例如行动太大、提醒不合适 |
| 下一步是什么 | Today 只展示这一步 |

## 6. 今日行动

今日行动不是普通任务，必须绑定关键条件。

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| title | 是 | 用户今天要做的动作 |
| linked_condition | 是 | 它补齐哪个条件 |
| done_when | 是 | 做到什么算完成 |
| minimum_step | 是 | 低门槛启动动作 |
| fallback_action | 是 | 状态差时的替代动作 |
| estimated_minutes | 是 | 预计耗时 |
| checkin_question | 是 | 完成后怎么反馈 |

## 7. 未完成诊断

没完成时，AI 不能只说“明天继续”。必须诊断两层问题。

第一层：行为为什么没发生。

| 类型 | 判断方向 | 调整 |
| --- | --- | --- |
| motivation | 用户并不真的想做 | 重审目标、暂停或换目标 |
| ability | 动作太大、太难、太抽象 | 缩小到最小步骤 |
| prompt | 提醒时间、渠道、话术不对 | 调整提醒 |

第二层：路径是否设计错。

| 类型 | 判断方向 | 调整 |
| --- | --- | --- |
| path | 行动没有补齐关键条件 | 重建行动 |
| condition | 当前缺口判断错了 | 改 current_gap |
| goal | 目标本身还没澄清 | 回到 Agent 澄清 |

## 8. 复盘

复盘不是统计打卡，而是回答：

```text
目标是否更清楚了？
哪个条件被补齐了？
哪个 KR 有进展？
哪个行动无效？
下一周期最该推进什么？
```

复盘结果默认写入 Logs 的 Markdown 文件。

## 9. 验收标准

| 编号 | Given | When | Then |
| --- | --- | --- | --- |
| AC-F1-1 | 用户输入模糊目标 | Agent 回答 | 先追问成功标准，不直接生成完整任务清单 |
| AC-F1-2 | 用户确认目标 | 系统生成目标推理卡 | 必须包含 KR、必要条件、充分条件、当前缺口 |
| AC-F1-3 | 系统生成今日行动 | 保存行动 | 行动必须绑定 linked_condition |
| AC-F1-4 | 用户连续未完成 | Agent 诊断 | 必须区分行为问题和路径问题 |
| AC-F1-5 | 一周结束 | 系统生成周志 | 复盘必须说明补齐了什么条件和下周推进什么 |
