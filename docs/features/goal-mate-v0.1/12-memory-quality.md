# F11 Memory Quality 规格

## 1. 模块定位

Goal Mate 的 Memory / Logs / Review / Meta-Cognition 不能沉淀 AI 式废话。

系统只应该沉淀：

```text
充分、必要、因果明确、语言清晰、可验证或可证伪的判断。
```

记忆不是知识堆积，而是假设管理。每条核心记忆都应该能影响后续目标、行动、风险提示、提醒时机或复盘判断。

## 2. 充分性

记录必须足以支撑后续判断。

不合格：

```text
用户今天状态不好。
```

合格：

```text
用户在固定行动窗口后仍未完成，反馈原因是“启动不了”，且过去 3 天同类行动完成率低。当前更可能是行动启动成本过高。
```

## 3. 必要性

记录必须对后续决策有影响。

不合格：

```text
用户说今天下雨。
```

合格：

```text
用户因外部环境阻碍未完成行动，因此明天需要准备可执行的替代动作。
```

如果一个信息不会影响目标、行动、风险、提示、复盘或元认知，就不应该成为核心记忆。

## 4. 因果性

系统不能只记录相关性，必须说明可能的因果链。

不合格：

```text
用户晚上容易失败。
```

合格：

```text
用户在某个高风险时段容易失败，当前因果假设是：风险点出现前没有替代动作，导致默认高风险行为发生。
```

如果因果关系只是猜测，必须写成“当前假设”，并说明后续怎么验证。

## 5. 清晰语言

记录必须使用清晰、准确、可操作的语言。

不合格：

```text
用户内驱力存在波动。
```

合格：

```text
用户连续 3 天没有完成核心行动，但每天都完成低成本记录。当前更可能是核心行动启动成本高，而不是整体放弃目标。
```

能说清楚的就说清楚；不能说清楚的，不要装作已经知道。

## 6. 可验证和可证伪

重要判断必须能被后续反馈证明、修正或推翻。

不合格：

```text
用户潜意识里抗拒目标。
```

合格：

```text
当前假设：用户没有完成能力训练任务，主要原因是任务过大。如果明天改为最低成本版本后仍不开始，就需要重新判断是否为方向问题或提示问题。
```

## 7. 核心记忆格式

核心记忆至少包含：

| 字段 | 说明 |
| --- | --- |
| `claim` | 清晰判断 |
| `scope` | 适用范围 |
| `evidence` | 事实依据 |
| `causal_explanation` | 因果解释 |
| `decision_impact` | 会影响什么后续决策 |
| `verification_signal` | 如何验证或证伪 |
| `confidence` | 当前置信度 |

## 8. 禁止沉淀

| 类型 | 原因 |
| --- | --- |
| 空泛鼓励 | 不影响后续决策 |
| 情绪标签 | 容易误判用户 |
| 无依据人格判断 | 不可验证且容易伤害用户 |
| 单次事件过度概括 | 证据不足 |
| 与目标推进无关的信息 | 增加记忆噪音 |
| 无法证伪的解释 | 不能进入反馈控制 |

## 9. 对 Logs / Review / Memory 的要求

| 模块 | 要求 |
| --- | --- |
| Logs | 记录观察、反馈、偏差和调整，不写无用流水账 |
| Review | 输出条件变化、路径判断、下一周期假设和验证方式 |
| Memory | 保存会影响后续干预的用户模式和目标模式 |
| Meta-Cognition | 保存关于 AI 自己判断是否有效的假设 |

## 10. 验收标准

| 编号 | Given | When | Then |
| --- | --- | --- | --- |
| AC-F11-1 | Agent 写入核心记忆 | 保存前 | 必须包含 evidence、causal_explanation、decision_impact 和 verification_signal |
| AC-F11-2 | Review 生成长期判断 | 输出时 | 必须说明依据和下一次验证方式 |
| AC-F11-3 | 日志中出现模糊描述 | 系统压缩为记忆 | 不得保存为核心记忆，除非转成可验证判断 |
| AC-F11-4 | AI 判断用户动机不足 | 输出或沉淀 | 必须标记为假设，并给出证据和验证方式 |
| AC-F11-5 | 记忆不会影响后续决策 | 质量检查 | 不得进入核心记忆层 |
