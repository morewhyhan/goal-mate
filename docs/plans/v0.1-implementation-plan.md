## 2. 目标

本 Plan 的目标是把 v0.1 需求规格逐条转成可执行开发任务，确保第一版不是普通 Todo、打卡工具或玩具聊天页，而是能跑通单主目标完整推进闭环的 Web Console。

v0.1 完成后必须成立：

```text
用户输入目的
-> Agent 澄清目标和成功标准
-> 系统生成目标推理卡、KR、条件、阶段计划
-> Today 展示唯一下一步
-> 用户反馈完成或未完成
-> Agent 诊断原因并调整路径
-> Logs 沉淀 Markdown 推进记录
-> Goals 只读展示目标结构和进度
-> Settings 能真实配置模型、提醒、日志和隐私
```

## 3. 非目标

- 不做移动 App、小程序、微信/飞书机器人完整接入。
- 不做自部署 MCP Server 完整实现。
- 不做多人协作、企业 OKR、复杂项目管理。
- 不做复杂日历、邮箱、外部自动执行。
- 不做知识库、MOC、灵感箱、复杂标签系统。
- 不把原型 HTML 当作需求真相；原型只作为表现参考。

## 6. 技术方案

开发顺序遵守工程规范：

```text
需求规格
-> 技术规格
-> 数据模型
-> API 契约
-> React Query hooks
-> 页面实现
-> Agent 结构化输出
-> 验收测试
```

v0.1 使用当前 `src` 模板的工程基础，但要移除模板里的 Tasks 示例业务。保留 Next.js App Router、Hono、Prisma、React Query、shadcn/ui、Tailwind 的分层方式。

模块分层：

| 层 | 职责 |
| --- | --- |
| Prisma | 保存目标、KR、条件、行动、日志、对话、设置、模型配置 |
| Hono API | 提供 typed RPC 接口，做鉴权、校验、权限和错误处理 |
| Hooks | 封装每个资源的数据读取和 mutation |
| Goal Engine | 处理目标推理卡、今日行动、诊断、复盘等领域规则 |
| Agent | 将自然语言对话转成结构化输出草案，并按确认边界落库 |
| Pages | 只展示必要信息和真实操作，不承载复杂业务推理 |

## 10. 页面推进顺序

| 顺序 | 页面 | 为什么先后这样排 |
| --- | --- | --- |
| 1 | Settings | 先确定模型、日志、提醒、隐私配置，否则 Agent 和 Logs 没有边界 |
| 2 | Logs | 先有记录层，后续 Today 和 Agent 才能沉淀结果 |
| 3 | Goals | 先展示目标结构，Today 才知道下一步来自哪里 |
| 4 | Today | 接上 Goals 的 current_focus_goal 和 DailyAction |
| 5 | Agent | 最后接入结构化推理和控制能力，避免先做成纯聊天玩具 |

说明：视觉原型可以并行调整，但数据链路实现应按上面顺序推进。

## 11. 数据推进顺序

| 顺序 | 实体 | 原因 |
| --- | --- | --- |
| 1 | User、UserSetting、ModelConfig | 所有数据归属、模型和权限基础 |
| 2 | Goal、GoalReasoningCard、KeyResult、GoalCondition | 目标推进核心 |
| 3 | StagePlan、DailyAction、Checkin、Diagnosis、Review | 每日推进与反馈闭环 |
| 4 | LogEntry | 记录层和 Markdown 自动写入 |
| 5 | AgentThread、AgentMessage | 对话历史和结构化输出追踪 |
| 6 | IntegrationAccount、ExternalActionRequest | 后续预留，不阻塞 v0.1 |

## 12. API 推进顺序

| 顺序 | API | 最小能力 |
| --- | --- | --- |
| 1 | `GET /settings`、`PUT /settings`、`POST /settings/models/test` | 配置读取保存和模型测试 |
| 2 | `GET /logs/tree`、`GET /logs/:id`、`PUT /logs/:id` | 日志树和编辑 |
| 3 | `GET /goals`、`GET /goals/:id` | 目标结构读取 |
| 4 | `GET /today`、`POST /today/checkin` | 今日行动和反馈 |
| 5 | `GET /agent/threads`、`POST /agent/threads`、`POST /agent/threads/:id/messages` | 对话历史和消息发送 |
| 6 | `POST /agent/structured-output/confirm` | 用户确认后落库目标、设置、日志或行动变更 |

## 14. 风险与回滚

| 风险 | 表现 | 处理 |
| --- | --- | --- |
| 先做页面导致核心链路空心化 | 页面好看但不能承载目标推理 | 先补 Schema、数据模型、API，再做交互 |
| Agent 做成普通聊天 | 只有自然语言，没有结构化输出 | 每个关键输出必须通过 schema 并保存草案 |
| Logs 变成知识库 | 出现 MOC、灵感箱、标签系统 | 坚持记录服务行动，只保留推进记录 |
| Settings 变成假开关 | 用户不知道开关影响 | 每项配置必须有当前值和影响说明 |
| 模板业务残留 | Tasks 页面和示例数据污染产品 | PLAN-001 必须先完成 |
| 多目标复杂度提前爆炸 | v0.1 被拉成完整版 | 只允许一个 current_focus_goal 进入 Today |

回滚策略：每个阶段单独提交。若某阶段失败，只回滚该阶段，不回滚 PRD 和规范文档。
