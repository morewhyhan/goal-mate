# 老三论控制闭环融入计划

## 1. 目标

把 `docs/shit/laosanlun-goal-mate.md` 中定义的系统论、信息论、控制论要求，落到 Goal Mate 的文档、Agent 行为、日志沉淀和验收门禁中。

本计划不追求在一轮内做完整商业产品，而是先确保 v0.1 的最小目标推进控制闭环真实成立。

## 2. 最终用户感知标准

用户不应该感知到“老三论”术语。

用户应该感知到：

```text
1. 它知道现在只推进哪个目标，不会把所有事搅在一起。
2. 它会先问清楚缺失信息，而不是立刻乱列计划。
3. 它安排的今日行动能对应一个当前关键缺口。
4. 我反馈完成或没完成后，它会真的更新判断。
5. 我没做时，它会判断是动机、能力、提示还是路径问题。
6. 日志能还原今天原本要做什么、实际发生了什么、系统为什么调整。
7. 下一次对话会基于最新状态继续推进，而不是重新开始聊天。
```

## 3. P0 必须闭环

```text
用户目标
  -> 目标边界
  -> 信息缺口
  -> 成功标准
  -> 关键条件
  -> 当前缺口
  -> 今日行动
  -> 执行反馈
  -> 偏差诊断
  -> 干预调整
  -> 日志/记忆沉淀
  -> 下一轮继续推进
```

## 4. 第一批改动

| 状态 | 改动 | 目的 |
| --- | --- | --- |
| done | 新增本计划 | 把理论转成可执行任务 |
| done | 更新 F1 系统控制规格 | 增加用户感知验收标准 |
| done | 强化 Agent system prompt | 让 Agent 每次对话先判断信息缺口、偏差和下一步干预 |
| done | 强化 Check-in 日志格式 | 让日志能还原观察、反馈、偏差和调整 |

## 6. 验收方式

第一阶段使用静态门禁验证：

```bash
pnpm verify:agent-loop:static
```

门禁必须覆盖：

```text
Agent system prompt 是否包含控制闭环规则。
Check-in 日志是否记录观察、反馈、偏差和调整。
F1 规格是否明确用户感知标准。
```

第二阶段再按需要做 runtime 验证。
