# Goal Mate v0.1 验收执行 Runbook

## 1. 目的

本 Runbook 用于执行 `PLAN-025`：按 v0.1 验收矩阵验证主链路是否成立。当前验收状态见 `docs/plans/v0.1-acceptance-status.md`。它不是新的需求来源，需求来源仍然是 `docs/features/goal-mate-v0.1/` 和 `docs/test-cases/goal-mate-v0.1-test-cases.md`。

## 2. 执行前置条件

| 条件 | 说明 |
| --- | --- |
| 用户授权 | 需要明确允许执行迁移、seed、类型检查、构建、测试或预览命令 |
| 数据库策略确认 | 当前模板迁移来自 Ignite，执行前需要确认是 reset 还是保留旧迁移追加 |
| 环境变量 | `DATABASE_URL`、Better Auth 相关 URL；模型 API Key 由用户在 Settings 中按账号配置 |
| 依赖完整 | `src/node_modules` 或 workspace 依赖已安装 |
| 当前代码已保存 | 工作区变更明确，不混入无关临时改动 |

## 3. 推荐验收顺序

```text
1. Prisma schema 检查和 client 生成
2. 数据库迁移或 reset
3. seed demo 数据
4. TypeScript / build 检查
5. API 契约验收
6. 页面主链路验收
7. PRD 验收矩阵逐条判定
8. 记录缺陷并回补 Plan
```

## 4. 命令清单

以下命令需要用户授权后执行。

### 4.1 Prisma generate

```bash
cd src
pnpm prisma generate
```

目的：确认 `src/prisma/schema.prisma` 可以生成 Prisma Client。

通过标准：命令成功，生成的 client 包含 Goal Mate v0.1 模型。

### 4.2 数据库迁移

当前不建议直接无脑执行旧迁移追加，因为旧迁移来自 Ignite 模板任务系统。建议二选一：

| 策略 | 命令 | 适用情况 |
| --- | --- | --- |
| reset | `pnpm db:reset` | 本地开发库，没有需要保留的数据；非交互执行，并跳过自动 seed |
| dev migration | `pnpm prisma migrate dev --name goal_mate_v01` | 需要保留迁移历史并生成新迁移 |

通过标准：数据库结构与 `schema.prisma` 一致。

### 4.3 Seed demo 数据

```bash
node scripts/seed-goal-mate.mjs
```

目的：创建 demo 用户、目标推理卡、KR、条件、阶段、今日行动、日志、Agent 对话、DeepSeek 模型配置和设置。

通过标准：脚本输出 seeded 成功，数据库中有 `demo@goalmate.local` 相关数据。

### 4.4 类型检查或构建

优先执行：

```bash
pnpm exec tsc --noEmit
```

如果项目未配置独立 typecheck，可执行：

```bash
pnpm build
```

通过标准：无 TypeScript 编译错误。若出现 Hono RPC 类型错误，优先修 hooks 的路径类型。

### 4.5 启动本地服务

```bash
pnpm dev
```

通过标准：服务启动，Web 可访问。

注意：不要在 `pnpm dev` 正在运行时并行执行 `pnpm build`。Next 的 `.next` 产物可能被生产构建覆盖，导致运行中的 dev server 找不到 chunk。若已经执行 build，需要重启 dev server 后再做 API 或页面运行时验收。

### 4.6 准备登录态验收用户

```bash
pnpm verify:v01:prepare-auth
```

目的：通过真实 Better Auth 接口注册或登录验收用户，并把 seed 用户的 Goal Mate 业务数据绑定到这个登录用户。

通过标准：输出 `Moved seed data: goals=1, logs=1, threads=1` 和 `GOAL_MATE_COOKIE='...'`。

### 4.7 业务主链路验收

```bash
GOAL_MATE_COOKIE='<cookie>' pnpm verify:v01:business:write
```

目的：验证 Settings、Models、Goals、Today、Logs、Agent、Checkin、Diagnosis、Review、Export 主链路。

通过标准：脚本输出所有 `BF-*` 项均为 `PASS`，并写入 `docs/plans/v0.1-business-flow-last-run.md`。

## 5. API 验收步骤

| 步骤 | API | 断言 |
| --- | --- | --- |
| 1 | `GET /api/health` | 返回 `product: goal-mate` |
| 2 | `GET /api/settings` | 登录后返回 General/Goals/Logs/Today/Agent/Notifications/DataPrivacy |
| 3 | `GET /api/models` | 返回 DeepSeek 默认模型，API Key 脱敏 |
| 4 | `GET /api/goals` | 返回 demo 目标、KR、条件、阶段和今日行动 |
| 5 | `GET /api/today` | 返回 current focus goal 和唯一 action |
| 6 | `POST /api/today/checkin` | 创建 Checkin，partial/not_done 创建 Diagnosis，LogEntry 追加 Markdown |
| 7 | `GET /api/logs/tree` | 返回日志 path 列表 |
| 8 | `POST /api/reviews/generate` | 创建 Review，并写入周期日志 |
| 9 | `GET /api/settings/export` | 导出数据，模型密钥不明文出现 |

## 10. 标准化执行入口

`src/package.json` 已提供以下验收入口：

| 脚本 | 用途 |
| --- | --- |
| `pnpm verify:v01` | 执行 Prisma generate 和 TypeScript 检查 |
| `pnpm verify:v01:api` | 对运行中的本地服务执行 API smoke checks |
| `pnpm verify:v01:api:write` | 执行 API smoke checks 并写入 `docs/plans/v0.1-acceptance-last-run.md` |
| `pnpm verify:v01:prepare-auth` | 创建登录态验收用户，并把 seed 业务数据绑定到该用户 |
| `pnpm verify:v01:business` | 执行登录态业务主链路验收 |
| `pnpm verify:v01:business:write` | 执行业务主链路验收并写入 `docs/plans/v0.1-business-flow-last-run.md` |

完整主链路验收仍需要先完成数据库迁移、seed 和本地服务启动。

## 11. 登录态验收模式

普通 `pnpm verify:v01:api` 只在没有 `GOAL_MATE_COOKIE` 时执行未登录安全 smoke，不等于完整主链路验收。

完整 API 验收必须提供登录态 cookie：

```bash
GOAL_MATE_COOKIE='<cookie>' pnpm verify:v01:api:auth
```

需要写入报告时使用：

```bash
GOAL_MATE_COOKIE='<cookie>' pnpm verify:v01:api:auth:write
```

如果未提供 `GOAL_MATE_COOKIE`，auth 模式必须失败，避免把未登录 smoke 误判为完整验收通过。
