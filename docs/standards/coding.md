# 编码规范

## 功能开发顺序

每个新功能必须按这个顺序推进：

1. 定义用户目的：一句话说明这个功能让用户完成什么。
2. 定义充分必要信息：删掉不影响理解和使用的一切内容。
3. 定义数据模型：先确定实体、字段、关系和权限边界。
4. 定义 API：用 Hono RPC 暴露最小必要接口。
5. 定义 hooks：用 React Query 封装读取、创建、更新、删除。
6. 实现页面：页面只组合组件和状态，不堆业务逻辑。
7. 更新文档：同步修改设计、接口、数据和验收说明。

## 命名规则

- 数据实体用业务名：Goal、KeyResult、LogEntry、AgentThread、ModelConfig。
- API 路由用复数资源名：`goals`、`logs`、`agent-threads`。
- React hooks 用 `use<Resource>` 或 `use<Resource>Mutation`。
- Query key 用数组：`['goals']`、`['goal', goalId]`、`['logs', period]`。
- 文件名使用 kebab-case：`goal-timeline.tsx`、`use-goals.ts`。

## 代码组织

- 一个模块的 API、hooks、组件和页面要能互相对应。
- 组件超过一个明确职责就拆分。
- 组件 props 必须表达业务语义，不传含义不明的布尔值。
- 页面文案必须能让用户一眼知道“这是什么、能做什么、现在该怎么用”。
- 不使用假按钮；按钮必须有真实状态、真实行为或明确禁用原因。

## 复杂度控制

- 先默认任何内容都是非必要的，再证明它必须存在。
- 不把 OKR、甘特图、日志、回顾等方法论原样压给用户。
- 用户页面展示结果，Agent 和系统内部处理复杂推导。
- 原型可以使用静态数据，但结构必须像真实产品数据。

## 错误处理

- 后端抛出统一错误，不在路由里散落随意字符串。
- 前端 mutation 失败必须给用户可理解的反馈。
- 空状态必须告诉用户当前为什么为空，以及下一步能做什么。

## 禁止项

- 禁止复制模板项目的业务无关文案。
- 禁止为了“看起来完整”加入无法操作的装饰面板。
- 禁止在页面里堆长段解释文字代替直觉设计。
- 禁止让用户滚动整个聊天页面才能找到输入框。
