# First-run Empty State

## 1. 原则

第一次建库、用户没有任何目标/日志/对话时，页面不能展示假任务、假 KR、假阶段、假甘特条或贴合某个具体故事的硬编码内容。

空状态只回答一个问题：

```text
用户现在下一步该做什么？
```

## 2. 首次使用顺序

```text
1. 用户登录。
2. 用户进入 Settings，配置模型 API Key。
3. 用户进入 Agent，用自然语言说明想达到的结果；如果信息不足，Agent 只追问一个关键问题。
4. Agent 生成目标草案，并生成“确认激活为当前目标”的待确认动作。
5. 用户确认激活目标。
6. Goals 只读展示 Objective / KR / Conditions / Stages / Daily Actions。
7. Today 只显示今天唯一下一步。
8. Check-in 或复盘后，Logs 自动沉淀 Markdown 推进记录。
9. 用户可以稍后在 Settings 配置 QQ Bot，生成一次性绑定码并在 QQ 中发送绑定命令，让系统主动追问和提醒。
```

## 3. 页面空状态

| 页面 | 无数据时显示 | 不允许显示 |
| --- | --- | --- |
| Today | “先告诉 Agent 你想达到什么结果”以及 Settings / Agent 入口 | 假四象限任务、假完成标准、假替代动作 |
| Goals | “等待真实目标生成”以及生成后会出现的结构说明 | 假 OKR、假 KR、假甘特条 |
| Logs | “完成一次 Check-in 或复盘后自动写入” | 示例日志内容伪装成真实记录 |
| Agent | 首次目标输入模板 | 空泛宣传文案、无操作入口 |
| Settings | 模型、QQ、提醒、部署状态的真实配置表单 | 不生效的假开关 |

## 4. 加载态

加载态必须和空状态区分。

```text
正在读取数据 != 没有数据
```

页面读取 API 时显示 loading 文案；只有 API 返回真实空数组或空对象时，才显示 first-run empty state。

## 5. 可验证标准

在全新数据库、只创建账号但不创建业务数据时：

```text
Today 不出现任何任务卡片。
Goals 不出现任何 KR 或甘特条。
Logs 不出现任何示例日志正文。
Agent 可以直接输入第一句话并自动创建对话。
Settings 可以配置模型和 QQ Bot。
```

## 6. 弹性状态

页面不能只区分“有数据 / 没数据”。至少要区分以下状态，并给出不同的最小下一步。

| 状态 | 用户真实处境 | 页面下一步 |
| --- | --- | --- |
| 缺模型 | 用户还没有配置 AI 能力 | 去 Settings 配置模型 |
| 有模型但无目标 | Agent 可以工作，但还不知道用户要什么 | 去 Agent 说明结果、截止时间、当前情况 |
| 有目标但无今日行动 | 目标结构存在，但没有落到今天 | 去 Agent 生成今天唯一下一步 |
| 有今日行动且未反馈 | 系统已经给出控制输出 | 只显示当前行动和反馈按钮 |
| 今日已反馈 | 今天的控制闭环已完成 | 显示已反馈状态，不重复催用户提交 |
| 无日志 | 还没有产生推进证据 | 不要求用户管理日志；引导去 Agent / Today |
| QQ 未配置 | Web 可用，但主动推送不可用 | 在 Settings 配置 QQ Bot；不阻塞目标创建 |

## 7. 从零到一闭环

| 步骤 | 用户看到 | 系统动作 |
| --- | --- | --- |
| 配模型 | Settings 首次设置卡片显示模型为下一步 | 保存用户模型 API Key，服务端加密 |
| 说目标 | Agent 首屏允许自然语言输入，不要求填表 | 无目标时先判断信息是否足够 |
| 信息不足 | Agent 只问一个关键问题 | 不创建假目标、假 KR、假今日行动 |
| 生成草案 | Agent 回复目标草案摘要 | 具体自然语言目标会触发 `goal.create_draft`，创建目标、KR、条件、阶段、今日行动和目标 Markdown；有模型 Key 时优先由模型路由生成，无模型 Key 时才使用本地兜底 scaffold |
| 确认激活 | Agent 动作卡显示待确认 | 创建 `goal.update` 待确认动作 |
| 查看目标 | Goals 出现 OKR / Gantt | 只读展示真实目标数据 |
| 执行今天 | Today 出现唯一下一步 | 用户点击完成/部分/没做/改小 |
| 沉淀记录 | Logs 出现 Markdown | Check-in / Review 自动追加推进证据 |
| 主动推进 | Settings 配置 QQ 后生成绑定码 | Worker 校验“绑定 GM-XXXXXX”后绑定当前账号的 QQ 会话 |

## 8. 设计规则

```text
业务数据不存在时，不用示例数据填满页面。
能力缺失时，先引导补能力。
结构缺失时，引导 Agent 生成结构。
行动缺失时，引导 Agent 生成今日行动。
行动存在时，页面只保留执行和反馈。
```

这种设计允许用户从不同入口进入系统：

```text
先配模型 -> 再定目标
先写目标草稿 -> 再补模型
先有目标 -> 再接 QQ
只用 Web -> 后续再接机器人
```
